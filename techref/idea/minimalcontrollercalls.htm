Minimal Controller Subroutine Call and Return

To understand how function calls work, we have to start from the last thing and go backwards in time. We know the goals in advance:

  • Stop executing the current program and switch to executing the subroutine program.
  • Pass some values or addresses (arguments) from the main program scope to the subroutine scope (where they become parameters).
  • Do some stuff in the subroutine.
  • When that finishes, return a value to the calling code's scope.
  • Resume executing the next part of the main program.

Note we want to do that from multiple different places in the main program, thereby re-using the function code. And the function might call another function and so on. So we use the stack to save what we need, so that we can do it multiple times. For example, we can push the arguments onto the stack before we switch to running the function.

That's the obvious stuff. Now let's start at the end and work back. When the function ends, it needs to:

  • Get back to the point in the main program where we need to pick up running again after the function is done. So that's one thing we need to save. We can pop that off the stack, assuming it's been saved there for us when we called the function.
  • But it also needs to clean all those arguments, so we need a count of how many to remove.
  • Finally, we need to somehow return a result.

So here is some example ABC code:

a:b(c,3)+2.

Call the function "b" with the value of "c", and 3 and then put the result in "a" and add 2 to it.

Let's say that b was defined as: b:"a*b."

Or something like that. Note that in the "b" function, "a" now points to the first arguments on the stack and "b" to the second (the 3). These are the arguments. We don't actually change the value of "a", but when we see a reference to it, we point there instead of to its real address. We do that for as many letters as there were arguments.

The key is that "." at the end. Here is what it has to do:

  1. Pop the return address and hold on to it so we can set the PC to it to restart execution of the main program. The ")" knows that value (it's the character after the ")" so easy for it to push it right before it sets the PC to the source address: "b").
  2. Pop the argument count and decrement the SP by that much to clear off the arguments. ")" knows that as well.
  3. Put the result back on the stack for the main program to pickup. Note this isn't a push, because we don't change the stack pointer. We just put it where we would if we were doing a push, then we fail to increment SP, and… we set the source to point at it. So when the main code runs "do_it" that value gets put into "a" (the real "a").
  4. But there is one more thing… to understand that, we need to go back to the start and run it forward again.

Let's also review the main code:

  • "a" sets the destination.
  • ":" is the operation (copy).
  • "b" is the source.
  • "(" Hmmm… Doesn't seem like this has anything to do, but at least it moves us from source to argument list… more on that in a second.
  • "c" sets the… wait a minute, we already have a destination and source… not to mention an operation. So the "(" needs to save that. It packs up the source (one of 26 with a flag for pin), the operation, and the destination. It could be 3 pushes, but it could also pack into a single value pretty easily.
  • "," takes the destination address and looks up its value and pushes it. It also increments the argument count.
  • "3" builds n, and then "," finds no destination, so it takes n and pushes it, and adds another argument to the count.
  • ")" pushes the argument count, and then the current PC + 1 as the return address, and then sets the PC to the saved original source (the "b")... and off we go.

There is also an opportunity for a shortcut in the form of a quick call to a function without parameters with something like "a"). If the parameter count is zero, ")" can just "do it all":

  • Save the current destination, op, and count on behalf of "(", knowing that "(" wasn't used.
  • Push a zero argument count.
  • Push the current PC + 1.
  • Set the PC to the source or destination (if there is no source).

It's nice to be able to call a function with only two tokens. The problem with it is that if the programmer writes e.g. "a()" then the "(" function would be done twice. There really doesn't seem to be a way to know. Maybe something clever like setting the count to -1 and then "(" sets it to 0, giving itself away.
